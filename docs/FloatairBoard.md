# FloatairBoard Environment

Chinese version: [FloatairBoard_cn.md](FloatairBoard_cn.md)

`FloatairBoard` documents the real-board development environment, ARM build inputs, resource preparation, and build artifacts for the glasses app layer. For the desktop simulator environment, see [FloatairSimulator.md](FloatairSimulator.md).

## 1. Scope

This document covers the `jy_app` ARM firmware app-layer target:

- Target platform: `arm`
- Platform adapter directory: `bes28/`
- Build target name: `uimain`
- Build entry: repository root `CMakeLists.txt`
- Filesystem packaging script: `scripts/fs_img.py`

Host protocol fields are documented in [datapath_v3_protocol.md](datapath_v3_protocol.md).

## 2. Base Tools

| Tool | Requirement | Description |
| --- | --- | --- |
| CMake | 3.10 or later | Minimum version required by the top-level `CMakeLists.txt` |
| Python 3 | Executable interpreter | CMake finds `Python3::Interpreter` for product overlay, UI resource generation, and filesystem packaging |
| ARM GCC | `arm-none-eabi-gcc` | ARM builds support GCC only. Clang is not supported. |
| ARM binutils | `arm-none-eabi-nm`, etc. | `scripts/fs_img.py` uses `arm-none-eabi-nm` for symbol checks |
| Python littlefs package | `littlefs` | `scripts/fs_img.py` imports `LittleFS` from `littlefs` to generate LittleFS images |
| genromfs | Optional | Used to generate ROMFS when available; otherwise the script uses its built-in Python ROMFS writer |
| 7-Zip | `7z` / `7za` | Used by CMake to extract the OS SDK archive when `JY_APP_OS_SDK_ARCHIVE` is provided |

`arm-none-eabi-gcc` can be placed in `PATH`, or passed explicitly during CMake configuration with `-DCMAKE_C_COMPILER=<path-to-arm-none-eabi-gcc>`.

## 3. Project Configuration

### 3.1 `.config`

The ARM platform reads `.config` from the repository root and converts its entries into compile definitions. ARM configuration fails if `.config` is missing.

After changing Kconfig options, rerun the full CMake configure step instead of only doing an incremental build.

### 3.2 Target Platform Detection

`cmake/TargetPlatform.cmake` detects the target platform from the compiler:

- `arm-none-eabi-gcc` -> `arm`
- If no compiler is specified and `arm-none-eabi-gcc` is in `PATH`, `arm` is selected by default
- To build the desktop simulator, specify the corresponding simulator compiler. See [FloatairSimulator.md](FloatairSimulator.md).

### 3.3 Product Overlay

`JY_APP_PRODUCT` selects the product overlay. The default value is `jytek`. During CMake configuration, the overlay is cleaned first and then the selected product overlay is applied.

`JY_APP_PRODUCT=clean` only removes overlay files and cannot produce a build.

### 3.4 OS SDK Archive

`jy_app` no longer keeps OS-synchronized `floatair`, `nuttx`, and `vendor` headers directly in the app repository. Both ARM and simulator builds consume an OS SDK archive exported from the OS repository.

Pass the archive on first configure:

```bash
cmake -S . -B build \
  -DCMAKE_C_COMPILER=arm-none-eabi-gcc \
  -DJY_APP_OS_SDK_ARCHIVE=/path/to/jy_os_sdk_<branch>_<tag>_<count>_g<hash>_dev.7z
```

CMake computes the archive SHA256, extracts it into `.os_sdk_cache/<short-sha256>/os_sdk/`, and records the full SHA256 beside the cache. Later configure runs can omit `JY_APP_OS_SDK_ARCHIVE`; CMake will reuse the newest valid cache. If no archive is passed and no valid cache exists, configuration fails.

The SDK package contains:

| SDK directory | Purpose |
| --- | --- |
| `os_sdk/floatair/` | Floatair headers shared by ARM and simulator builds |
| `os_sdk/nuttx/` | NuttX headers required by the ARM include path |
| `os_sdk/vendor/` | Vendor headers required by the ARM include path |
| `os_sdk/manifest.json` | Package manifest generated by the OS repository script |

`cmake/OsSdk.cmake` also generates build-directory override headers under `os_sdk_overrides/` to inject the app log prefix `FLA>` without editing the extracted SDK cache.

## 4. Resource Preparation

| Directory / File | Purpose |
| --- | --- |
| `romfs/` | Firmware read-only resources. ARM builds copy it into build-directory staging before packaging. |
| `lfsc/` | LittleFS C partition source directory. The built `uimain` is copied here for packaging. |
| `lfsd/` | LittleFS D partition source directory. Generates `nuttx_lfsd.bin`. |
| `StringPool.csv` | Multi-language string pool input. Packaging generates i18n JSON. |
| `ui.res.json` | UI resource description input |
| `apps/**/*.ui.json` / `system/**/*.ui.json` | UI compiler inputs. Outputs are generated under build-directory `generated/ui`. |

## 5. Startup Logo

The startup logo uses a static image:

| Item | Requirement |
| --- | --- |
| Path and file name | `/jyt_d/apps/logo/logo.png` |
| Image format | PNG is recommended. PNG / JPG / BMP are supported. |
| Recommended size | `240x80` |
| Maximum size | Must not exceed the optical display size `640x480` |
| Background | Transparent or pure black |
| Graphic elements | Pure white or pure green |

## 6. Build Flow

Core ARM build flow:

1. CMake detects the `arm` platform and loads `cmake/PlatformArm.cmake`.
2. Sources are collected from `apps/`, `system/`, `common/`, `lvgl/`, `thirdparty/`, and generated UI output.
3. The `uimain` ELF is generated.
4. After build, `romfs/` is copied into build-directory staging.
5. `scripts/fs_img.py --source <uimain> --romfs-dir <romfs_staging>` is executed.
6. Filesystem images are generated.

Common CMake configuration:

```bash
cmake -S . -B build \
  -DCMAKE_C_COMPILER=arm-none-eabi-gcc \
  -DJY_APP_OS_SDK_ARCHIVE=/path/to/jy_os_sdk_<branch>_<tag>_<count>_g<hash>_dev.7z
cmake --build build
```

If `PATH` already contains `arm-none-eabi-gcc`, `CMAKE_C_COMPILER` can be omitted. If a valid `.os_sdk_cache/` entry already exists, `JY_APP_OS_SDK_ARCHIVE` can also be omitted.

## 7. Build Artifacts

After an ARM build, the build directory contains:

| Artifact | Description |
| --- | --- |
| `uimain` | ARM app-layer ELF |
| `nuttx_lfsc.bin` | LittleFS C partition image |
| `nuttx_lfsd.bin` | LittleFS D partition image |
| `nuttx_romfs.bin` | ROMFS image |
| `generated/` | Generated UI code, resource headers, build config, and related generated content |

## 8. Common Issues

| Symptom | Check |
| --- | --- |
| CMake cannot find the ARM compiler | Make sure `arm-none-eabi-gcc` is in `PATH`, or pass its full path through `-DCMAKE_C_COMPILER` |
| CMake reports missing `.config` | Make sure `.config` exists in the repository root |
| CMake reports missing OS SDK cache | Pass `-DJY_APP_OS_SDK_ARCHIVE=<path-to-jy_os_sdk_..._dev.7z>` once, or keep a valid `.os_sdk_cache/` entry |
| CMake cannot find 7-Zip | Install 7-Zip or set `-DJY_APP_OS_SDK_SEVEN_ZIP=<path-to-7z>` |
| `fs_img.py` cannot import `littlefs` | Install the `littlefs` package for the active Python environment |
| Symbol check fails | Check whether a source file is missing, config macros are correct, and referenced symbols are allowed |
| ROMFS generation fails | Check whether `romfs/` exists and whether its content exceeds the partition limit |
| LittleFS generation fails | Check whether `lfsc/` or `lfsd/` exceeds the partition capacity |

## 9. Board vs Simulator

| Item | Board | Simulator |
| --- | --- | --- |
| Target platform | `arm` | `linux` / `macos` / `mingw` / `msvc` |
| Platform directory | `bes28/` | `simulator/FloatairSimulator/` |
| Main artifact | `uimain` and filesystem images | `floatair_simulator` / `floatair_simulator.exe` |
| Graphics output | Real optical display | SDL2 window |
| Resource path | Packaged into LittleFS / ROMFS | `jyt_d/` / `romfs/` under the build directory |
